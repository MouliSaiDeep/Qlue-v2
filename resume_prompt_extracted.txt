3.2 Status Value Case Mismatch
Backend uppercase vs Frontend lowercase status values
 [CRITICAL]
 — The backend writes status values as UPPERCASE strings (PENDING, UPLOADING, PARSING, PARSED, FAILED) while the frontend checks for lowercase strings ("parsed", "parsing", "failed"). This means the frontend will never correctly identify a resume's status.
Affected files:
• 
Backend: processResumeUpload.js writes 'PARSING', 'PARSED', 'FAILED'; generatePresignedUrl.js writes 'UPLOADING'; resume.js model defaults to 'PENDING'
• 
Frontend: resume_detail_screen.dart line 99-101 checks `resume.status == "parsed"`, `resume.status == "parsing"`, `resume.status == "failed"`
• 
Frontend: mock_data.dart defines mock resumes with lowercase status strings
Fix: Normalize status values at the API boundary. Either convert backend responses to lowercase before sending to the frontend, or update the frontend to handle uppercase status values. The recommended approach is to add a status normalization step in the frontend API service layer that converts backend uppercase values to lowercase.
3.3 processResumeUpload Dual-Identity Bug
S3 trigger handler exposed as REST API endpoint
 [CRITICAL]
 — The processResumeUpload handler is written as an S3 ObjectCreated event trigger (it reads event.Records[0].s3.object.key), but the SAM template.yaml defines it as a POST /resume/process API endpoint. When called via the API Gateway REST endpoint, event.Records will be undefined and the handler will crash with a TypeError.
Fix: Decide on a single invocation pattern. Two options:
1. 
Keep as S3 trigger: Remove the API endpoint from template.yaml, add an S3 EventNotification property on the ResumesBucket to trigger this Lambda directly on ObjectCreated events. The current code will work as-is.
2. 
Convert to REST API: Rewrite the handler to accept resumeId in the request body, then look up the S3 key from the database record and process accordingly. This is the more flexible approach for local development.
3.4 createResume Status Override Bug
Model overwrites UPLOADING status to PENDING
 [HIGH]
 — In resume.js model, the createResume() function always sets status: 'PENDING', overriding the 'UPLOADING' status that generatePresignedUrl.js passes in. This means resumes will never be in the UPLOADING state, breaking the upload status tracking flow.
Fix: In resume.js, change the createResume function to respect the status passed in resumeData, only defaulting to 'PENDING' if no status is provided:
async function createResume(resumeData) {
    const item = {
        ...resumeData,
        status: resumeData.status || 'PENDING',  // Respect passed status
        uploadedAt: Date.now(),
        isActive: false
    };
    return await put(RESUMES_TABLE, item);
}
3.5 Resume Limit Mismatch
Frontend allows 5 resumes, backend allows 4
 [MEDIUM]

Fix: Standardize on a maximum of 5 resumes (4 stored + 1 buffer). Update backend generatePresignedUrl to check `>= 5` instead of `>= 4`, and update getResumeList to return maxAllowed: 5.
3.6 AWS Region Mismatch
DynamoDB vs S3/Textract region inconsistency
 [MEDIUM]
 — The dynamodb.js file has conflicting region definitions: the v3 client uses 'us-east-1' while the legacy v2 client uses 'ap-south-1'. The s3.js and textract.js both use 'ap-south-1'. The bedrock.js uses 'us-east-1'. Cross-region calls will fail or have increased latency.
Fix: Standardize all AWS clients to a single region. If using Bedrock (which is only available in us-east-1 for some models), ensure all services use us-east-1, or configure cross-region access properly.
3.7 Unused Libraries in Resume Handlers

5.3 Fix Express Handler Wrappers
All resume handlers are written as Lambda handlers expecting event/requestContext format. When imported into Express, they receive (req, res) instead. Currently, the Express routes directly pass the handler functions, which will fail because Express passes different arguments. Fix by creating adapter wrappers in server.js:
function lambdaToExpress(handler) {
  return async (req, res) => {
    const event = {
      body: JSON.stringify(req.body),
      pathParameters: req.params,
      queryStringParameters: req.query,
      requestContext: { authorizer: { uid: req.user?.uid } },
    };
    const result = await handler(event);
    res.status(result.statusCode).json(JSON.parse(result.body));
  };
}

7. Key Architecture Decisions
The following decisions should be made before implementation begins:
REST API vs S3 Trigger for processResumeUpload: 
Recommend REST API pattern. This allows the frontend to trigger processing after confirming the S3 upload completed, enables local development without S3 event notifications, and gives better error feedback to the user. The S3 trigger approach would require polling from the frontend anyway to know when processing completes.
Status normalization strategy: 
Recommend frontend-side normalization. Add a normalizeStatus() function in the API service layer that converts uppercase backend status values (PARSED, PARSING, FAILED, UPLOADING, PENDING) to lowercase before creating ResumeModel objects. This keeps the backend consistent (uppercase is the DynamoDB convention) and isolates the conversion to a single layer.
Route unification target: 
Recommend standardizing on the SAM template.yaml route patterns (singular /resume/*). The frontend api_constants.dart already matches these routes. Only the Express server.js needs updating.
State management for resume data: 
Use the existing Provider pattern (consistent with AuthProvider and InterviewProvider). The ResumeProvider should be the single source of truth for all resume data in the app.
